## 1. TRIVIAL WRAPPERS — "The Triplet Problem" — RESOLVED

**Immutables removed: `setModelCy/getModelCy` and `setZHalf/getZHalf` deleted** (commit 50143a4)

`MODEL_CY` (always 0) and `Z_HALF` (always 1) were immutable constants with unnecessary getter/setter pairs. Values inlined at 4 call sites, 4 files deleted, `zoomState.zHalf` default updated to 1, dead `Z_HALF` removed from `viewportState`.

**Mutable properties (zoom, W, H) remain as-is** — the one-function-per-file pattern is justified for properties that change at runtime with independent update frequencies (e.g., W/H change on resize, zoom changes on input). Consolidated setters would require read-modify-write, adding caller weight for no gain.

## 4. CONCEPTUAL OVERLAP

**Triangle fill: 3 separate implementations**

| File | Role |
|------|------|
| `setFillTriangleOnLayer.js` | fills on the fill layer canvas (with alpha blending) |
| `setFillTriangle.js` | fills for the worker rasterizer |
| `setRenderTriangle.js` | combines fill + wire in one pass |

The PURPOSE docs claim these serve different pipelines, but the core logic (parse color → fill path → stroke) is copy-pasted across all three. The "unified renderer" (`setRenderMeshUnified`) was clearly bolted on later as a simpler alternative, but the original multi-layer path was never decommissioned.

**Projection: 2 implementations with different names**

| File | Role |
|------|------|
| `utilProjectedVertices.js` | single-point projection for UI/debug |
| `utilTransformSync.js` | full vertex transform + projection as fallback |

Both use the **same projection formula** (`screen = center + (world * fov) / (worldZ + 3)`) but it's defined in two separate files with slightly different parameter names. The formula isn't shared.

## 5. NAMING SMELLS

| File | Smell |
|------|-------|
| `stateVertexTransformBridge.js` | Not a bridge. It's a state container for the worker. "Bridge" implies two-way translation. |
| `stateFillWorker.js` | Describes what the **worker** does (triangle sorting, lighting), not what the **state** holds. |
| `setRenderForegroundState.js` | Name says "set" but it's really a **state container** (holds the function pointer). The actual setter is `setRenderForeground.js`. |
| `stateLoop.js` | Has `setRunFrame` in its PURPOSE text — cross-contamination of concerns. |
| `setPostToBackgroundWorker.js` | "Post" implies async message. The actual function is `worker.postMessage` with a null check — calling it a "set" is misleading. |

## 6. ARCHITECTURAL CONCERNS

### 6a. State explosion

There are **12+ state files** in `engine/state/render/` alone. Each wraps a single concern (canvas elements, viewport, physics, zoom, input canvas, etc.). The getter/setter pairs add 20+ more files. This creates an import tax — modules that need W, H, MODEL_CY, and Z_HALF must import from 4 separate state modules instead of one.

### 6b. init/mesh is 33 files

The mesh initialization pipeline is spread across 33 individual files. Some are legitimate (parseOBJ, parseFace, parseNormal), but others are one-line helpers that could live in their parent module:

- `initComputeMaxRadius.js` — computes `Math.max(...vertices.map(length))`
- `initClampToUnitSphere.js` — `Math.min(radius, 1.0)`
- `initCheckRadiusLayer.js` — checks one grid shell

These are testable helpers, yes, but at the cost of **file density** that makes navigation harder.

### 6c. set* functions that should be init* or build*

| File | Actually does |
|------|---------------|
| `setClusterDecimate.js` | does decimation (an init/build operation, not a state setter) |
| `setCapModelForCpu.js` | creates a decimated model (building, not setting) |
| `setSeedParticles.js` | creates particle arrays (building, not setting) |

The "set" prefix implies state mutation, but these functions do significant computation. Callers looking for "where does decimation happen?" won't find it under `set*`.

## 8. RECOMMENDATIONS (Priority Order)

### High Priority

1. **Merge the zoom/viewport setters into stateful setter modules**
   - `setZoom({ zoom, min, max })` and `setViewport({ W, H, MODEL_CY, Z_HALF })`
   - Eliminates ~8 files with zero functionality loss
   - Reduces import tax across the codebase

2. **Unify the triangle rasterization**
   - 3 implementations of essentially the same loop
   - The unified renderer (`setRenderMeshUnified`) should be the only one
   - The 3-layer compositing path can be retired

### Medium Priority

3. **Rename set* files that are actually build/init operations**
   - `setClusterDecimate` → `buildDecimatedMesh`
   - `setCapModelForCpu` → `initCpuCapModel`
   - `setSeedParticles` → `buildParticleArray`

4. **Consolidate duplicate canvas visibility functions**
   - Merge `setCanvasCpuHidden.js` and `setCanvasHidden.js` into one

5. **Fix naming mismatches**
   - `stateVertexTransformBridge.js` → `stateVertexTransformWorkerState.js` (or similar)
   - `setRenderForegroundState.js` → move to `state/` properly
   - `setPostToBackgroundWorker.js` → `postToBackgroundWorker.js` (drop the "set")

6. **Unify projection math**
   - Extract the projection formula into a shared `util` function
   - Both `utilProjectedVertices.js` and `utilTransformSync.js` should import it

### Low Priority

7. **Fill in missing WHY sections** (3 files)
8. **Reconsider init/mesh file density** — consider grouping related helpers (e.g., `initMeshMath.js` for `computeMaxRadius`, `clampToUnitSphere`, `computeBoundingBox`)
